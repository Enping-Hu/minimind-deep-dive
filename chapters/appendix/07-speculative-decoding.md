# 延伸：投机解码——不改输出分布地让生成变快

主线里 MiniMind 的生成是标准逐 token 自回归：`eval_llm.py` 调 HF 的 `model.generate` + KV cache（[04-inference/01](../04-inference/01-kv-cache-and-generate.md) 讲过 KV cache 怎么让每步只算当前 token）。每生成一个 token 走一遍模型，`web_demo.py` 的 `TextStreamer` 一个字一个字往外蹦，这是最直观、也最慢的解码方式。

这一篇讲一个能让它明显变快、却**不改变输出分布**的方法：投机解码。它没有 MiniMind 源码可钉，但能回答一个推理提速里最反直觉的问题：**为什么用一个小模型"猜"，反而能让大模型的输出和单独跑时一模一样、还更快？**

一句话主线：**逐 token 解码慢在每步都要把整个模型权重搬一遍显存、却只算一个 token；投机解码用便宜模型先猜几个、再让大模型一次并行验证，把"搬权重"的成本摊到多个 token 上，且配对接受规则后输出严格无损。**

## 先看清瓶颈：decode 是带宽受限，不是算力受限

自回归生成每步只产出一个 token。关键事实是：这一步通常**先被显存带宽限制，而不是先被算力限制**。每生成一个 token，都要把巨大的权重从显存搬一遍，但真正参与计算的只有「一个 token 那一列」，GPU 的算力大量闲着。

这就留出了空子：既然走一遍模型的成本几乎被「搬权重」主导、和「这一遍顺手多算几个 token」关系不大，那能不能**一次 forward 多吐几个 token**？投机解码就是冲这个浪费来的。

（对照着看会更清楚：[量化](06-quantization.md) 省的是「每步搬运量」，投机解码省的是「decode 步数」——两者正交，可叠加。）

## 核心直觉：先猜后验

投机解码分三步：

- **draft（起草）**：用一个**便宜**的模型 `q` 先自回归猜出 `γ` 个候选 token。它便宜，所以猜 `γ` 步的代价远小于让大模型走 `γ` 步。
- **verify（验证）**：把这 `γ` 个候选一次性喂给**目标大模型** `p`，在**一次并行 forward** 里同时算出每个位置的分布。这一步和 prefill 一样能吃满 GPU，代价却和单走一步几乎一样。
- **接受/纠正**：从前往后比对，接受最长的一段「大模型也认可」的前缀；在第一个被否决的位置，由大模型重采一个纠正 token；后面没用上的候选直接丢掉，进入下一轮。

一轮下来，**最好情况**是 `γ` 个候选全被接受、再白送一个 bonus，一次大模型 forward 吐出 `γ+1` 个 token；**最坏情况**是第一个就被否，那也至少拿到 1 个正确 token，退化成普通逐 token 解码，不会更慢（除了多花的那点 draft 成本）。

## 为什么输出和大模型单独跑严格一致

投机解码最容易被误解成「用小模型近似、牺牲质量换速度」。**不是**。只要验证用的是 speculative sampling（Leviathan et al. / Chen et al., 2023）那套接受规则，输出分布和「只用大模型逐 token 采样」**严格相同**，是 lossless 的。这一点和量化那种有损压缩有本质区别。

接受规则是一种改良的拒绝采样。对 draft 在某位置采到的 token `x`，设 draft 给它的概率是 `q(x)`、target 给它的概率是 `p(x)`：

- 以概率 $\min\!\big(1,\,p(x)/q(x)\big)$ **接受** `x`；
- 一旦**拒绝**，就从「被低估的残差分布」里重采一个 token，即从归一化后的 $\big(p(x)-q(x)\big)_+$ 中采样。

可以证明这样得到的 token 恰好服从 target 分布 `p`。直觉上：draft 高估的部分按比例打折扣，draft 低估、漏掉的部分在拒绝时补回来，两边一加正好填平成 `p`。所以投机解码**不需要在质量和速度之间权衡**，这是它和绝大多数推理优化最不一样的地方。

（一个常见简化：如果 draft 和 target 都用贪心 argmax，验证就退化成「token 是否相等」，实现更省，但此时匹配的是 greedy 轨迹、而非完整采样分布。）

## 能快多少：接受率说了算

设每个位置 draft 被接受的概率为 `α`（接受率），它衡量 **draft 和 target 有多对齐**。一轮里 target 一次 forward 平均能产出的 token 数是一个等比和：

$$E_{\text{out}} = \frac{1-\alpha^{\gamma+1}}{1-\alpha}$$

这个期望介于 1 与 `γ+1` 之间：`α→1` 时趋于 `γ+1`，`α→0` 时趋于 1（普通逐 token 解码只有 1）。真实墙钟加速还要扣掉 draft 成本，所以划算的条件是两条同时成立：

- **draft 足够便宜**：draft 走 `γ` 步 + target 验证 1 次，总成本要明显小于 target 走 $E_{\text{out}}$ 步；
- **`α` 足够高**：draft 和 target 要经常在「下一个 token 该是什么」上一致，否则猜了也白猜。

`γ` 不是越大越好：太大，后面那些大概率被拒的候选就在浪费 draft 算力；太小，又没充分利用「一次并行验证」的便宜。实践里 `γ` 常取 `3~8`，并随接受率动态调整。结论很朴素：**要么 draft 极便宜、要么 draft 极准，至少占一头才有正收益。**

## draft 从哪来：几条路线

投机解码的工程差异，大半在「`q` 从哪来、怎么和 `p` 对齐」：

| 路线 | draft 怎么产生 | 特点 / 代价 |
|------|---------------|-------------|
| 独立小 draft 模型 | 同家族的小模型（如 7B 配 70B） | 实现直接；要求 tokenizer/vocab 与 target 一致，小模型本身要够准 |
| self-draft 多头 | 在 target 上加几个轻量预测头，一次并行出多个 token（Medusa） | 不需要第二个模型、省显存；头要额外训练 |
| 特征级自回归 | 在 target 的隐藏特征层面起草再投影回 token（EAGLE） | 接受率高、draft 极轻；实现更侵入，要接 target 内部表示 |
| 检索 / 复制 | 从 prompt、KV 历史或语料里**查**出可能的续写（prompt lookup、REST） | 零额外模型、零训练；只在重复/可检索文本上接受率高 |

还有一条和训练耦合的路线：**MTP（Multi-Token Prediction）**。DeepSeek-V3、GLM4.5 在预训练时就加了 MTP 头预测后续多个 token，这些头训练时是辅助目标（提升表示质量），推理时正好拿来当 draft，相当于把「起草能力」内化进主模型，省掉独立 draft 的对齐烦恼。

## token tree：一次验证多个候选

线性起草只押一条轨迹，中途一被拒后面再准也用不上。**token tree** 同时押多条：让 draft 在每个位置给出 top-k 候选、构成一棵候选树，再用 **tree attention**（带掩码的注意力，让每个候选只看见自己那条根到叶的路径）在一次 forward 里并行验证整棵树，最后接受最长的被认可路径。SpecInfer、Medusa 的树版属于这一类。代价是树越宽、单次验证越重，要在「更高接受率」和「更重的验证」之间平衡。

## 什么时候真有用

投机解码不是无条件加速，边界很清楚：

| 场景 | 收益 | 原因 |
|------|-----|------|
| 低 batch / 延迟敏感（聊天、Agent 单请求） | **大** | decode 本就带宽受限、GPU 闲着，并行验证几乎免费 |
| 高 batch / 吞吐优先 | **小甚至为负** | 大 batch 下 GPU 已被喂到算力受限，draft 和验证在和真实请求抢算力 |
| 文本高度可预测（代码补全、模板、RAG 续写） | **大** | 接受率 `α` 高，n-gram 起草就够 |
| 高熵、强创造性输出 | **小** | draft 难猜准，`α` 低 |

几个落地必须处理的细节：**tokenizer 对齐**（draft 和 target 必须共享 vocab，否则逐 token 比对无从谈起）；**KV cache 回滚**（被拒候选写进 cache 的部分要能干净丢弃、回到接受点）；**变长调度**（每轮产出 token 数不定，batching 调度器要能处理「有的请求吐 5 个、有的吐 1 个」）。

## 落回 MiniMind

MiniMind 用的是最朴素的逐 token 生成，没有投机解码。放进这条线看，它正好是那个「未优化基线」：每步一个 token、搬一遍权重、GPU 大量闲着。

真要给 MiniMind 加投机解码，最省事的路线是「独立小 draft 模型」——它有现成的多档尺寸（如 26M 配更大的 MoE 版），只要 tokenizer 一致（MiniMind 全系共用一个 tokenizer，天然满足）就能配对。但也要清醒：投机解码在**低 batch、延迟敏感**时收益最大，而 MiniMind 这种小模型本身单步就快、搬权重的绝对成本低，投机解码的收益不如在大模型上明显。它更适合作为「读懂大模型推理系统怎么提速」的概念样本，而非 MiniMind 本身的必备优化。

## 常见误区

- **「投机解码用小模型近似，会掉质量」**——不。配 speculative sampling 接受规则后输出分布与纯 target 采样严格一致，是 lossless 的。
- **「draft 猜错了就慢了」**——不会更慢。最坏情况退化成逐 token 解码，仍至少拿 1 个正确 token，只多花一点 draft 成本。
- **「投机解码在任何场景都提速」**——不。高 batch 吞吐场景下 GPU 已算力受限，draft/验证反而抢资源，可能负收益。
- **「它和量化二选一」**——不。量化省每步搬运量、投机解码省 decode 步数，两者正交、可叠加。

## 练习

1. decode 阶段的瓶颈通常是带宽还是算力？投机解码正是利用了这个瓶颈的什么特点？
2. 「先猜后验」三步分别是什么？最好和最坏情况各产出几个 token？
3. 为什么说投机解码是 lossless 的？接受/拒绝规则怎么保证输出仍服从 target 分布？
4. 接受率 `α` 和候选数 `γ` 各影响什么？为什么 `γ` 不是越大越好？
5. 为什么投机解码在低 batch 延迟敏感场景收益大、高 batch 吞吐场景可能负收益？

1. 通常是显存带宽（bandwidth-bound）：每步要搬一遍权重却只算一个 token，GPU 算力闲着。投机解码利用「一次 forward 顺手并行验证多个 token 的额外算力几乎免费」这一点，把搬权重成本摊到多个 token 上。
2. draft（便宜模型猜 γ 个）→ verify（大模型一次并行 forward 验证）→ 接受/纠正（接受最长认可前缀 + 1 个 token）。最好情况 γ 全接受 + 1 个 bonus = γ+1 个；最坏情况第一个就被拒 = 1 个。
3. 因为接受规则是改良拒绝采样：以 min(1, p(x)/q(x)) 接受，拒绝时从残差分布 (p−q)₊ 归一化后重采。draft 高估的按比例打折、低估漏掉的在拒绝时补回，合起来正好填平成 target 分布 p，所以严格无损。
4. α 衡量 draft 与 target 对齐程度、决定平均每轮接受多少；γ 是每轮猜的候选数。γ 太大则靠后候选大概率被拒、浪费 draft 算力，γ 太小则没充分利用一次并行验证，所以有个最优区间（常 3~8），并随 α 动态调。
5. 低 batch 时 decode 带宽受限、GPU 闲着，并行验证几乎免费，收益大；高 batch 时 GPU 已被 continuous batching 喂到算力受限，draft 和验证要和真实请求抢算力，可能得不偿失。
