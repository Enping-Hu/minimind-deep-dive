# 延伸：模型量化——把 26M 塞进手机之前要懂的事

主线没有量化代码：MiniMind 训练用 BF16 混合精度（`train_pretrain.py:81` 的 `--dtype bfloat16`），权重以 `.pth` 存成 FP 格式，[04-inference/03](../04-inference/03-weight-formats.md) 也只把 GGUF / MNN 这类端侧格式「点到为止」。但只要你想把模型真正部署（塞进手机、跑在 CPU、或让显存放下更长的上下文），量化几乎是绕不开的一步。GGUF / MNN 之所以能在端侧跑，核心就是量化。

这一篇讲清量化的三组旋钮（位宽、粒度、阶段）和 LLM 量化真正的难点。它没有 MiniMind 源码可逐行钉，但能回答一个部署时最先撞上的问题：**同一个模型，为什么能从几百 MB 压到几十 MB 还基本能用？代价是什么？**

一句话主线：**量化是用位宽换体积和速度的有损压缩，做得好不好，在 LLM 上几乎都取决于怎么对付激活里的少数离群值。**

## 先分清：BF16 不是量化，但和量化是一个话题

MiniMind 训练用 BF16，这是个好起点，因为它和量化共享同一套直觉：**用更少的位表示一个数，省内存、可能更快，但要付精度的代价。**

- `FP32`：32 位，最常用的高精度，训练的默认参照。
- `FP16`：16 位，占用减半，但**动态范围**（能表示的最大/最小量级）比 FP32 窄，大数容易溢出。
- `BF16`：同样 16 位，却保留了和 FP32 一样宽的指数范围，牺牲尾数精度换动态范围。深度学习训练偏爱它，正因为训练里数值的**量级跨度**比小数点后的精度更要命——这也是 MiniMind 默认 `bfloat16`、而 `GradScaler` 只在 `float16` 时才启用的原因（`train_pretrain.py:130`：FP16 动态范围窄，需要 loss scaling 防梯度下溢，BF16 不用）。

但 BF16/FP16 仍是**浮点**。真正意义上的量化，是再往下一步：把浮点近似成**低位整数**（INT8、INT4），用离散格点表示原本连续的数值。这一步收益更大（INT8 只有 FP32 的 1/4 位宽），损失也更硬，从"精度变粗"变成"落到有限的整数格点上"。

## 量化在做什么：一个线性映射

核心是把浮点值 `x` 映射到整数 `x_q`：

$$x_q = \mathrm{clip}\big(\mathrm{round}(x / s) + z,\ q_{\min},\ q_{\max}\big)$$

`s`（scale）是量化间隔，决定每个整数格点代表多大的浮点跨度；`z`（zero-point）是零点偏移；`round` 取整、`clip` 截断到整数范围。两种基本形态：

- `z = 0` 是**对称量化**：推理时不算 `z` 相关项，算子更简单更快。
- `z ≠ 0` 是**非对称量化**：按数据实际的最小/最大值定范围，更充分利用整数区间、误差更低，代价是多一点计算。

理论上权重、激活分布不均匀，非等距格点（非线性量化）误差更小；但非线性量化推理时算起来贵，所以实践几乎都用等距格点的线性量化。

## 三组旋钮

量化效果由三根旋钮决定，理解它们就理解了所有量化方案在调什么。

**旋钮一：位宽。** INT8 是工业界最常用的甜点，低于 8 比特统称低比特量化。INT4 更激进、用于资源极度受限场景。极限是 1 比特二值网络，能把模型压到约 1/32、用位运算极快地算，但精度损失很大。位宽越低，压得越狠、损失越大。

**旋钮二：粒度——多少个数共享同一组 `(s, z)`。**

- **per-tensor（逐层）**：一整层共用一组，最简单。但一层里只要有个别极端值，就会把范围撑大、压低整体精度。
- **per-channel / per-token（逐通道）**：权重每列一组（per-channel）、激活每行一组（per-token），更细、更准、也更复杂。
- **per-group（逐组）**：每 K 行/列共用一组，介于两者之间。`groupsize=1` 退化成逐层，等于通道数时等价于逐通道。

越细越准越慢，这是精度和速度的直接权衡。

**旋钮三：量化对象。** 光量化权重只省体积；要真正拿到整数矩阵乘法的**加速**，权重和激活得**都**量化。长序列时 KV cache 还会膨胀到压过权重（[04-inference/01](../04-inference/01-kv-cache-and-generate.md) 讲过 KV cache 怎么涨），量化 KV cache 对长序列吞吐很关键，这一点正好接上前一篇 [KV cache 压缩](04-kv-cache-compression.md)：GQA/MLA 减少缓存条数，KV cache 量化降低每条的位宽，两条路互补。

## 在哪个阶段量化：PTQ vs QAT

按量化发生在训练流程的哪一步，分三档，对应不同的精度/成本权衡：

- **PTQ（训练后量化）**：训练完直接量化，只需少量校准数据、不重训。简单高效，是 LLM 量化的主战场。缺点是可能掉点。
- **QAT（量化感知训练）**：训练时就插入「伪量化节点」（前向先量化再反量化，人为引入量化误差），让模型在优化任务损失时一并适应低精度。精度最好，成本最高。
- **QAF（量化感知微调）**：只在微调阶段引入量化，压缩与保性能之间折中。`QLoRA` 就带这个思路——冻结 4bit 量化权重、只训外挂 LoRA 分支（[附录 LoRA 入口](01-advanced-pointers.md) 提过 LoRA 的冻结主干+低秩旁路）。

LLM 场景 PTQ 是主流，因为重训代价太高。而 PTQ 做得好不好，几乎全看下面这一件事。

## LLM 量化的真正难点：激活离群值

PTQ 在 LLM 上最棘手的问题是**激活离群值**：激活里少数特征维度的绝对值显著大于其余，且集中在很少几个维度上。无论 per-token 还是 per-channel，这些离群值都会把量化范围撑大、严重拉低其余正常值的精度。

有意思的是，**权重反而好量化**——权重可以在推理前预先算好量化系数；难的是激活，它的分布取决于具体输入、事先不知道。几个代表方法就是围绕「怎么对付离群值」分出的不同思路：

| 方法 | 核心思路 | 对付离群值的方式 |
| --- | --- | --- |
| `LLM.int8()` | 混合精度分解 | 把含离群值的少数维度拆出来用 FP16 高精度算，其余走 INT8 |
| `SmoothQuant` | 难度迁移 | 用平滑系数把激活的「难度」部分转移给权重，让两边都好量化 |
| `GPTQ` | 量化当优化问题 | 逐层逐参数找最优量化值，量化一个后调整同块内其余参数补偿误差 |
| `AWQ` | 激活感知 | 权重并非同等重要，保住约 1% 显著权重就大幅降误差；按激活幅度逐通道缩放权重 |

一条清晰的线索：`LLM.int8()` 和 `SmoothQuant` 用最简单的舍入到最近格点（RTN），靠预处理绕开离群值；`GPTQ` 把量化当优化问题、逐块求更优权重；`AWQ` 进一步指出关键不在「平均误差」而在「保住少数重要权重」。**LLM 量化的真正难点不是权重，而是激活离群值**——谁能在不牺牲速度的前提下把离群值处理好，谁的低比特量化就掉点更少。

## 落回 MiniMind

MiniMind 本身不量化，但这条线正好补上它部署故事的最后一环：

- 训练用 **BF16**（省训练显存、稳），这是浮点侧的"半步压缩"，还不是量化。
- 权重存成 FP 的 `.pth` / Transformers 目录，是训练和研究用的全精度形态。
- 真要上端侧（[04-inference/03](../04-inference/03-weight-formats.md) 的 GGUF / MNN），就进入整数量化的领域——GGUF 常见的 `Q4_K_M`、`Q8_0` 等后缀，正是「INT4/INT8 + 某种分组粒度」的具体组合。MiniMind 的 26M 模型 FP16 存约 50MB，量化到 INT4 能压到十几 MB，才好塞进手机或浏览器。

所以量化在 MiniMind 的技术版图里是**部署侧的下一步**：主线把模型训出来、存成 FP，量化负责把它压到端侧能承受的体积，代价是接受一点可控的精度损失。

## 常见误区

- **「BF16 就是量化」**——不。BF16/FP16 仍是浮点，只是精度更低；量化特指映射到低位**整数**格点。
- **「量化只压权重」**——不够。只量化权重省的是体积；要拿到整数运算加速，激活也得量化；长序列还要量化 KV cache。
- **「位宽越低越好」**——有代价。位宽越低压得越狠，但精度损失越大，INT8 是当前多数场景的甜点，INT4 及以下要靠 AWQ/GPTQ 这类方法才掉点可控。
- **「量化难在权重」**——反了。权重好量化（系数可预先算），难的是分布随输入变、且带离群值的激活。
